# シンプル化ルール

## 基本原則

- **YAGNI:** 今必要ない機能は作らない
- **KISS:** シンプルに保つ
- **過度な DRY より読みやすさ:** 少しの重複は OK

## 禁止パターン

- 1 回しか使わないヘルパー関数
- 「将来のため」のオプションや引数
- 使われていない引数
- 「パターンだから」という理由のファクトリー / ラッパー
- 必要のないクラス階層

## 抽象化の判断基準

**する:**

- 同じコードが 3 回以上登場
- テスト対象のロジック (pure / impure 分離のため)
- 明確なドメイン概念
- 依存性注入が必要 (テストでモック差し替えしたい等)

**しない:**

- 「将来使うかも」
- コードを短くするためだけ
- 「クラスにすると綺麗」だけが理由

## 既存コードの「歴史的経緯」を尊重

本リポジトリには 2020 年初出のコードと 2026 年追加のコードが混在する。
**気軽に「綺麗にしましょう」リファクタを始めない**:

- 既存ロジックは長期運用で角が取れている
- 副作用が多く、テスト無しでは挙動が読みづらい
- 触る範囲は要件と直接関係ある箇所のみに絞る

例: `apps/insta360_auto_converter.py` の `main()` は 300 行モノリスだが、
テストが薄いため一気に分解せず、新規モード (`local_auto_converter.py`) を
**並列に追加する** 形で対応している。

## 「3 回登場ルール」の運用

3 回目を書く瞬間に抽象化を検討する。**2 回目までは堂々とコピペ**:

- 1 回目: 必要なものを書く
- 2 回目: コピペする (ここで抽象化したくなるが、まだ早い)
- 3 回目: コピペが 3 回目になる瞬間に抽象化する (関数 / クラス / 共通ファイル)

## チェックリスト

- [ ] 今作成しているコードは本当に必要か?
- [ ] 「将来のため」に追加した引数 / 設定 / 抽象化はないか?
- [ ] このリファクタは要件と関係あるか? (関係ないなら別 PR で議論)
- [ ] 1 ファイル / 1 関数で完結する規模か? (大きいならアーキテクチャルールを再確認)

# TDD ルール

このファイルが TDD ルールの Single Source of Truth。
詳しい背景・モックポリシー・例外条件は `.kiro/steering/testing.md` を参照
(常時インクルージョンで読み込まれる)。

## 大原則

**今後の実装はすべて TDD (Test-Driven Development) で進める。**

- 機能追加 / バグ修正 / リファクタを問わず、まず失敗するテストを書く
- テスト無しで本実装をコミットしない

## サイクル

1. **Red — 失敗するテストを書く**
   - 仕様を表現する最小のテストを 1 つ書き、`make test` で **必ず失敗** することを確認する
   - 失敗の理由が「未実装」または「期待した分岐に入らない」であることをログで確認する

2. **Green — 最小実装で通す**
   - テストを通すためだけのコードを書く。過剰な汎化や設計の作り込みはしない
   - `make test` でそのテストが通ることを確認する

3. **Refactor — 設計を整える**
   - テストが緑のまま、命名・重複・構造を整える
   - リファクタの途中で赤になったら巻き戻すのではなく、小さく分割して再度赤→緑

## ディレクトリ

- テストはリポジトリ直下の `tests/` に配置
- ファイル名: `test_<対象モジュール>.py`
- 関数名: `test_<挙動>` 形式

## モックポリシー (要約)

- **外部サービス境界 (Google API / Insta360 SDK / subprocess)** は基本的にモックする
- **ファイル I/O は `tmp_path` fixture で実 I/O を回す** (実装乖離を避ける)
- **MediaSDK バイナリは絶対に呼ばない** (NDA、CI に置けない、決定論性ゼロ)

## 例外的に TDD を緩める場合

以下に限る:

1. **typo 修正・コメント変更・lint 自動修正など、挙動に影響しない変更**
2. **設定ファイルのみの変更** (`pyproject.toml` の dev deps 追加など)
3. **third-party バイナリ更新** (MediaSDK / ExifTool 等のバージョン上げ)

それ以外 (ロジック変更 / I/O 変更 / エラーハンドリング変更) は **必ず** TDD を踏む。

## CI ゲート

- `tests/` が存在する状態で、CI の Python ジョブが `uv run pytest` を必ず実行する
- テストが赤の PR は `main` にマージしない

## チェックリスト

PR を出す前に:

- [ ] 変更したコードパスに対応するテストを `tests/` に追加 / 更新したか?
- [ ] `make test` (= `uv run pytest`) が green か?
- [ ] テストが先に Red になることを確認したか? (実装してからテスト書いていないか?)
- [ ] 外部 API / SDK / subprocess を実呼び出ししていないか?
